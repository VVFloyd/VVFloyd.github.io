静态代理和jdk动态代理

java 动态代理
java
发布日期:   2021-03-30

静态代理

首先用代码模拟一个容易理解的现实场景，来解释到底什么是代理模式。
市面上流行的游戏主机有三种，PS,NS和XBOX，三足鼎立雄踞游戏市场。
这三个游戏机都支持最基础的游戏功能，但是随着发展出现了一些外部设备来增强这些游戏机的游戏体验。
这种增强就是在原游戏机主机的基础上，附加了很多新的功能，比如摄像头啊、健身环啊。
用代码表示这种关系，可以用装饰者模式，也可以用代理模式。这里我们用代理模式来实现这个关系。

模拟：

  1. 游戏机接口

    public interface GameStation {

 void getDesc();

 void getEquipment();
}

  2. 模拟PS

    public class PlayStation implements GameStation {

 @Override
 public void getDesc(){
     System.out.println("我是PS，我的游戏最多最好玩。");
 }

 @Override
 public void getEquipment() {
     System.out.println("主机");
 }
}

  3. 模拟NS

    public class NintendoSwitch implements GameStation{

 @Override
 public void getDesc(){
     System.out.println("我是次世代zui强的主机NS~");
 }

 @Override
 public void getEquipment() {
     System.out.println("主机");
 }
}

  4. 模拟PS的各种外设代理

    public class PlayStationProxy implements GameStation {

 private PlayStation playStation;

 PlayStationProxy(PlayStation pojo){
     this.playStation = pojo;
 }

 @Override
 public void getDesc() {
     System.out.println("自我介绍：");
     this.playStation.getDesc();
 }

 @Override
 public void getEquipment() {
     this.playStation.getEquipment();
     System.out.println("体感摄像头");
     System.out.println("VR");
 }
}

  5. 模拟NS的各种代理

    public class NintendoSwitchProxy implements GameStation {
 private NintendoSwitch nintendoSwitch;

 NintendoSwitchProxy(NintendoSwitch pojo){
     this.nintendoSwitch = pojo;
 }

 @Override
 public void getDesc() {
     System.out.println("自我介绍：");
     this.nintendoSwitch.getDesc();
 }

 @Override
 public void getEquipment() {
     this.nintendoSwitch.getEquipment();
     System.out.println("健身环");
 }
}

说明： 可见，代理类（4 PlayStationProxy，5 NintendoSwitchProxy）对被代理的对象（2 ps，3 ns）进行了功能的增强。getDesc方法增强都是多加了一句描述（增强内容相同），getEquipment方法增强了各自的外部设备（增强内容不同）。
这种增强的实现方式是代理类4 PlayStationProxy和被代理对象2 ps共同实现同一个接口1，代理类4 PlayStationProxy维持了一个2 ps的实例。当测试类调用时，调用代理类4 PlayStationProxy，就可以使用到被代理对象2 ps的方法，同时这个方法被增强了。

这就是代理模式，不对被代理类本身修改，而是通过实现同一个接口，对新的实现类（即代理类）进行修改增强。

以上这种实现方式就是静态代理。为啥叫静态的，因为对PS和NS分别需要两个代理类，即使他们的方法名是一样的，都是实现了同一个接口，增强的内容也是一样的。

JDK动态代理

动态代理有几个主要角色类，1.被代理类ps；2.被代理类ns; 3.生产代理对象的工具类ProxyUtil; 4.InvocationHandler的实现类
JDK动态代理和静态代理的区别就是，当增强的内容一样的时候，动态代理类可以一类多用，不需要为每个被代理类生产代理对象。
假如我们的需求仅仅是增强getDesc方法，那么我们需要下面几个类

3.生产代理对象的工具类ProxyUtil,这个方法获得了一个同样的代理类，可以对ps和ns代理

public class ProxyUtil {

    //绑定代理类和被代理类
    public static Object getProxyInstance(Object object){
        //新建代理类proxy,绑定被代理类object
        GameStationProxy proxy = new GameStationProxy(object);
        /**
         *         调用Proxy.newProxyInstance获取代理对象实例
         *         参数1：被代理类的类加载器
         *         参数2：被代理对象的所有接口
         *         参数3：定义了具体如何增强被代理类的代理类
         */
        return Proxy.newProxyInstance(object.getClass().getClassLoader(),object.getClass().getInterfaces(),proxy);
    }
}

4.InvocationHandler的实现类
生产通用代理类需要一个class com.sun.proxy.$Proxy0类，这个类定义了具体如何增强方法。这个类的创建仅需要实现InvocationHandler的invoke放法即可。
就是在invoke里定义了怎么增强被代理类的方法。

public class GameStationProxy implements InvocationHandler {
    //被代理类
    private Object object;

    GameStationProxy(Object o){
        this.object = o;
    }

    //代理类调用方法
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {

        if ("getDesc".equals(method.getName())){
            System.out.println("自我介绍：");
        }
        //调用被代理类的方法
        Object invoke = method.invoke(object, args);

        return invoke;
    }
}

测试方法：

public class Test {
    public static void main(String[] args) {
        //定义被代理ps
        PlayStation ps = new PlayStation();
        //获取代理类实例
        GameStation proxyInstance = (GameStation) ProxyUtil.getProxyInstance(ps);
        //调用代理类的方法
        proxyInstance.getEquipment();
        proxyInstance.getDesc();
    }
}

输出结果：
主机
自我介绍：
我是PS，我的游戏最多最好玩。

JDK动态代理的深入研究

如上我们就实现了动态代理，不论再来几种游戏机，不论是xbox还是小霸王，都不需要再新建代理类了。
但是又有个疑问，如果我相对PS和NS的getEquipment方法增强要怎么实现呢?他们增强的方式是不一样的。
ps是体感摄像头和VR，ns是健身环.

于是我对InvocationHandler的实现类的实现类进行改造

public class GameStationProxy implements InvocationHandler {
    //被代理类
    private Object object;

    GameStationProxy(Object o){
        this.object = o;
    }

    //代理类调用方法
    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        //getDesc增强
        if ("getDesc".equals(method.getName())){
            System.out.println("自我介绍：");
        }

        //getEquipment增强
        if ("getEquipment".equals(method.getName())){
            //增强
            if (proxy instanceof GameStation){
                System.out.println("体感摄像头");
                System.out.println("VR");
            }
            if (proxy instanceof NintendoSwitch){
                System.out.println("健身环");
            }
        }

        //调用被代理类的方法
        Object invoke = method.invoke(object, args);

        return invoke;
    }
}

这个写法挺想当然的，测试后发现并没有被增强。于是去看了看instanceof的实现方法：
boolean result;
if (obj == null) {
result = false;
} else {
try {
T temp = (T) obj;
result = true;
} catch (ClassCastException e) {
result = false;
}
}

这个方法的实现方式就是先判断 A instanceof B 中，A是不是空，如果是空，那A能不能强转为B。
强转异常，就捕获了返回false.顺带提一嘴，在很多业务中都可以用类似的结构。
或者对主流程不影响的代码块，也可以捕获但是不抛出。打个日志就好了。

看到这里就知道，是我对proxy的理解不对。我以为proxy 是ps ,ns ，即被代理类。
但是System.out.println(proxy.getClass().toString());打印发现其实是class com.sun.proxy.$Proxy0，即代理类。

那代理类到底是什么构成，用一个测试类来看看：

public class TempTest {

        public static void main(String[] args) throws Throwable{

            //这里指定被代理类
            GameStation rs=new PlayStation();
            InvocationHandler ds=new GameStationProxy(rs);
            Class<?> cls=rs.getClass();

            //以下是一次性生成代理

            GameStation subject=(GameStation) Proxy.newProxyInstance(
                    cls.getClassLoader(),cls.getInterfaces(), ds);

            //这里可以通过运行结果证明subject是Proxy的一个实例，这个实例实现了Subject接口
            System.out.println(subject instanceof Proxy);

            //这里可以看出subject的Class类是$Proxy0,这个$Proxy0类继承了Proxy，实现了Subject接口
            System.out.println("subject的Class类是："+subject.getClass().toString());

            System.out.print("subject中的属性有：");

            Field[] field=subject.getClass().getDeclaredFields();
            for(Field f:field){
                System.out.print(f.getName()+", ");
            }

            System.out.print("\n"+"subject中的方法有：");

            Method[] method=subject.getClass().getDeclaredMethods();

            for(Method m:method){
                System.out.print(m.getName()+", ");
            }

            System.out.println("\n"+"subject的父类是："+subject.getClass().getSuperclass());

            System.out.print("\n"+"subject实现的接口是：");

            Class<?>[] interfaces=subject.getClass().getInterfaces();

            for(Class<?> i:interfaces){
                System.out.print(i.getName()+", ");
            }

        }

}

输出结果：

true
subject的Class类是：class com.sun.proxy.$Proxy0
subject中的属性有：m1, m4, m2, m3, m0,
subject中的方法有：equals, toString, hashCode, getDesc, getEquipment,
subject的父类是：class java.lang.reflect.Proxy
subject实现的接口是：com.anonymous.template.abstract_factory.proxy_test.GameStation,

由此可见，class com.sun.proxy.$Proxy0继承了Proxy，实现了GameStation，真相大白。

但是可是，我们的问题依然没有解决，还是无法对getEquipment增强。
其实方法是有的，就是针对不同被代理类，创建不同的代理类。如果ps和xbox都是体感摄像头和vr,那他俩就公用一个代理类，ns是健身环，没人和他一样，他就单独用一个代理类。
这个实现方法看起来挺搓的，但是我查了一些解决方案都是这样。这种方法也是一种设计模式的实现原理，即抽象工厂模式。
