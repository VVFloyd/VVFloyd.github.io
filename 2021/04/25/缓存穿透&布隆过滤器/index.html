缓存穿透&布隆过滤器

java redis
java
发布日期:   2021-04-25

1. 缓存穿透

key对应的数据在数据源并不存在，每次针对此key的请求从缓存获取不到，请求都会到数据源，从而可能压垮数据源。比如用一个不存在的用户id获取用户信息，不论缓存还是数据库都没有，若黑客利用此漏洞进行攻击可能压垮数据库。

解决方法：缓存null值；布隆过滤器

1.1. 缓存null值

 @Override
    public void getNullFromDB() {
        //searchKey模拟入参
        String searchKey=null;
        Object o = redisUtil.get(searchKey);
        if(o!=null){
            //缓存有数据，返回
            return;
        }
        //随便找个表，模拟查询为空
        BArticle bArticle = bArticleMapper.selectByPrimaryKey(null);
        //不论是不是空，都放到缓存里。当然如果做得精细，null的缓存时间可以短一点
        redisUtil.set(searchKey,bArticle,1000);
        if (bArticle!=null){
            //其他业务
        }
    }

1.2. 布隆过滤器

1.2.1. 简单理解

布隆过滤器的原理是，把所有可能存在缓存里的key，存到布隆过滤器里。每次查询前先去过滤器里面看看，如果有，再去查询，没有直接就返回了。这样就能避免黑客制造一些一定不存在的key，去攻击缓存。缓存落空后就会查询数据库，从而起到攻击数据库的作用。

所以，如果不深究实现，可以假想布隆过滤器是一个set，set里有值，才去查数据执行逻辑，没值查都不差直接拒绝。

1.2.2. 数据结构

但是布隆过滤器显然不是用set,否则将造成大量的空间浪费。

布隆过滤器听起来高大上，但是本质上其实是一个bit数组（顺便回顾一下， java基本类型的长度 bit:位,一个二进制数据(0或者1) ），初始值全部都是0。

布隆过滤器的实现就是用特定的算法，把入参的key（举例是baidu），通过算法算出几个值（结果可能是12，888，2934），然后把这三个索引位置的值置为1.下次查询时，再传入baidu,分解后变成12，888，2934，过滤器去这三个位置一看，都是1，那么就认为baidu这个key是存在的。

1.2.3. 造成误判的原因

特别注意的是，布隆过滤器实现的功能实际上是可以用来告诉你 “某样东西一定不存在或者可能存在”。

过滤器说baidu不存在，那他一定不存在，但是他说baidu存在，其实不一定存在。为啥会这样呢，举个栗子：

  1. 数据库A表里有三个值：baidu,kugou,sougou.

  2. 我把这三个值放进redis里，同时放进布隆过滤器。

  3. 布隆过滤器存baidu,算法算出位置是12，888，2934，均置为1.

  4. 布隆过滤器存kugou,算法算出位置是65，777，988，均置为1.

  5. 布隆过滤器存sougou,算法算出位置是432，1777，2988，均置为1.

  6. 用户查询baidu,kugou,sougou都在过滤器里，ok没问题。

  7. 恶意用户查询tencent,布隆过滤器算法算出位置是12，777，3777.检查这前两个位置都是1，第三个是0，布隆过滤不通过，过滤正常。

  8. 恶意用户查询google,布隆过滤器算法算出位置是12，777，1777.检查这三个位置都是1，布隆过滤通过，误判。

    布隆过滤器的误差是由这个数组的大小决定的，数组越大，算法越精确，经过hash运算和重合的概率越小。过小的布隆过滤器很快所有的 bit 位均为 1，那么查询任何值都会返回“可能存在”，起不到过滤的目的了。布隆过滤器的长度会直接影响误报率，布隆过滤器越长其误报率越小。 即使过滤器不够大，对于这种恶意攻击也起到了一定的过滤作用，这种误差是可以接受的。

    看不懂的看这个https://zhuanlan.zhihu.com/p/43263751，有图，好顶赞。

1.2.4. 代码举例

<!--布隆过滤器-->
        <dependency>
            <groupId>com.google.guava</groupId>
            <artifactId>guava</artifactId>
            <version>25.1-jre</version>
        </dependency>
/**
 * 布隆过滤器测试类
 * @author vvf
 */
public class BloomFliterDemo {
    private static int size = 1000000;

    private static BloomFilter<Integer> bloomFilter = BloomFilter.create(Funnels.integerFunnel(), size);

    public static void main(String[] args) {
        for (int i = 0; i < size; i++) {
            bloomFilter.put(i);
        }
        //落在过滤器中的情况
        inBloom();
        //误判的情况
        outBloom();

    }

    private static void outBloom() {
        List<Integer> list = new ArrayList<Integer>(1000);
        // 故意取10000个不在过滤器里的值，看看有多少个会被认为在过滤器里
        for (int i = size + 10000; i < size + 20000; i++) {
            if (bloomFilter.mightContain(i)) {
                list.add(i);
            }
        }
        System.out.println("误判的数量：" + list.size());
    }

    private static void inBloom() {
        // 获取开始时间
        long startTime = System.nanoTime();
        //判断这一百万个数中是否包含29999这个数
        if (bloomFilter.mightContain(29999)) {
            System.out.println("命中了");
        }
        // 获取结束时间
        long endTime = System.nanoTime();
        System.out.println("程序运行时间： " + (endTime - startTime) + "纳秒");
    }

}

1.2.5. 移除元素

理论上是不允许的，因为每个位置可能被多个元素共用，删除一个可能会影响其他元素。当然解决方法也是有的。有以下几种解决方法：

1）对于关小黑屋再放出来这种需求，我们可以换一个思路，再加一个布隆过滤器。在A过滤器中存在的数据，再去B过滤器中瞅瞅，B中也有，那视为实际上没有。

2）如果是redis实现的布隆过滤器（Redis本身就支持bitmap这种数据结构，通过setbit和getbit即可实现一个Bloom Filter ），设个过期时间就可以删除对应的元素。这种实现类似于强行删除，感觉并不太好。

3）名为 Counting Bloom filter 的变种可以用来测试元素计数个数是否绝对小于某个阈值，它支持元素删除。可以参考文章 Counting Bloom Filter 的原理和实现 。CBF在本身数组上加了一位，用来计数。没次把指定索引位由0改成1时，count加1，每次删除时count减1.

结尾OS：布隆过滤器在项目中有用到，但是还是用的少，而且不久之后就被取缔了。一个是因为使用场景还是相对少，另一个是因为习惯上往往能通过结构优化解决的，就不用这个了。我也不知道这种对于技术的保守是谨慎负责还是矫枉过正ㄟ( ▔, ▔ )ㄏ
